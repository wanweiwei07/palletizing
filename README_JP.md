# パレタイジングプランナー

ロボットパレタイジング向けの三次元ビンパッキングソルバー。異なるサイズの箱が与えられた場合、配置位置と向きを計算し、パレット空間利用率を最大化します。

## ソルバー

### GA3D — GPU 高速化遺伝的アルゴリズム
- 染色体 = 箱の配置順序の順列 + 回転フラグ
- デコーダ = ハイトマップ + ビットパック ボクセル、貪欲 best-fit 配置 + 支持面制約
- GPU 上で集団を並列デコード（融合 Triton スキャンカーネル、染色体 1 つにつき 1 ブロック）
- 構造化初期化：集団の 2/3 を長さ降順にソート（半分はステップ `[0, n/2)` を回転、半分は `[n/2, n)` を回転）、残り 1/3 はランダム順列 + 交互回転

### CP-SAT — 制約プログラミング (fill3d)
- Google OR-Tools CP-SAT による厳密解法
- プローブベースの支持面積制約
- 小規模インスタンスに最適

### レイヤーベースプランナー (palletizing)
- 箱を高さ別にレイヤーにグループ化
- 各レイヤーで CP-SAT による二次元配置
- レイヤー間の支持面検証

## クイックスタート

### GA3D — 混合箱型 3D パッキング + 可視化
```bash
python visualize_ga3d.py 7      # task_70_seed7
```

### レイヤープランナー — 単一箱型パッキング
```bash
python main.py \
  --pallet-length 1200 --pallet-width 1000 --pallet-max-height 1500 \
  --box-length 300 --box-width 200 --box-height 250 --count 70
```

### タスク生成
```bash
python generate_task.py --count 70 --seed 7
```

## ベンチマーク

### task_70（混合箱型 70 個, pop=1024, 30s）

| Seed | 積載   | 体積利用率 |
|------|--------|-----------|
| 7    | 65/70  | 90.6%     |
| 42   | 67/70  | 87.0%     |
| 99   | 67/70  | 80.9%     |
| 123  | 70/70  | 85.3%     |

### task_lx（35 箱, パレット 1000×1200×1600, RTX 5090, seed=42）

| pop  | 30 秒         | 60 秒         | 1 世代あたり |
|------|---------------|---------------|-------------|
| 512  | 78.7% (88 世代)  | 78.7% (178 世代) | 0.34 秒     |
| 1024 | 78.1% (60 世代)  | 78.1% (120 世代) | 0.50 秒     |
| 2048 | 78.1% (35 世代)  | 78.1% (69 世代)  | 0.87 秒     |
| 4096 | 78.1% (18 世代)  | 78.7% (36 世代)  | 1.69 秒     |

全構成で 35 箱すべてを配置。体積利用率は 78.1–78.7% に集中。構造化初期化により、pop=512/30s でもこのタスクの上限に到達しています。

体積利用率 = 箱総体積 ÷ (パレット底面積 × 最上箱の頂部高さ)

## パラメータ調整ガイド

`GA3DConfig` のパラメータ（[ga3d/solver.py](ga3d/solver.py) 参照）：

| パラメータ | 既定値 | 効果 |
|-----------|--------|------|
| `pop_size`           | 2048  | 多様性 vs 1 世代あたりのコスト。大きいほど 1 世代は遅いが探索が広くなる。s/gen はほぼ線形。 |
| `time_limit_seconds` | 60.0  | 実時間予算。経過時間がこの値を超えると終了。 |
| `mutation_rate`      | 0.15  | 染色体ごとの swap 変異確率。上げると探索が増える。 |
| `crossover_rate`     | 0.8   | OX1 交叉確率。 |
| `elite_ratio`        | 0.1   | 毎世代そのまま引き継ぐエリートの割合。 |
| `tournament_k`       | 4     | トーナメント選択のサイズ。 |
| `min_support_ratio`  | 0.8   | 支持面割合の下限。0.9 にすると安全側、下げると密に詰む。 |
| `grid_res_xy/z`      | 10 mm | ボクセル解像度。小さいほど精密だが遅くなる（メモリ/計算は 2 乗/3 乗で増加）。 |
| `use_triton`         | True  | 融合 Triton スキャンカーネル。デバッグ時のみ無効化。 |
| `seed`               | None  | 指定すると再現可能。 |

**実用的な推奨：**
- 小規模 / 整ったタスク（≤ 50 箱）：`pop=512, time=30s` で十分。
- 中規模（50–100 箱）：`pop=1024–2048, time=30–60s`。
- 困難 / 混合度が高いタスク：`pop=2048–4096, time=60–120s`。
- 1 世代あたりのコストがボトルネックなら `pop_size` を下げる。早期収束するなら `pop_size` または `mutation_rate` を上げる。
- task_lx のスイープは平坦で、これはデコーダ / 評価関数の上限を示しており GA の限界ではありません。さらなる改善にはデコーダ側（空隙充填、より細かい支持面スコア）の改修が必要です。
